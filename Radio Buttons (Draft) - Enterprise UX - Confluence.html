Radio Buttons (Draft)

  • Created by Reichers, Pamela (Contractor), last modified by Bersofsky, Yvonne M. (Contractor) on Apr 03, 2025

USWDS 3 LEVEL 2: FULLY COMPLIANT

Please Note: The Enterprise UX Standards are required to be followed by NSF applications. If there are concerns regarding implementation, consult a member of the EUX Team.

Overview

Radio buttons are UI elements designed for selecting one option from a set of mutually exclusive choices. They are ideal when users need to make a single, clear selection from a list of options. Unlike checkboxes, which allow multiple selections, radio buttons ensure only one choice is active at a time, automatically deselecting the previous option when a new one is selected. Unlike dropdown menus or comboboxes, radio buttons display all available choices at once, reducing cognitive load and enabling easy comparison of options.

When to Use Radio Buttons

  • Best for mutually exclusive selections where only one choice is allowed.
  • Enhance user experience by making choices visually explicit and preventing confusion.
  • Use when space allows all options to fit in one column.
  • Ideal for 27 manageable options that can be displayed simultaneously.
  • Effective for simple binary decisions, such as Yes/No choices.

When to Use Other Elements

  • Checkboxes: When users can select multiple options, unselect all, or when theres only one item to select.
  • Dropdown Menus/Comboboxes: For lists with more than 7 options or when space is limited.

Usage and Best Practices

Labels

  1. Use Clear and Descriptive Labels

    • Keep labels concise, informative, and unambiguous (e.g., "Pick-up" instead of "Option 1").
    • Avoid vague, overly technical, or negative phrasing.

  2. Make Labels Selectable

    • Allow users to click or tap on either the label or the radio button to make a selection.
    • Ensure buttons are large enough for easy interaction (minimum 44x44px touch target).

  3. Ensure Consistent Formatting

    • Use uniform font size, style, and color for all labels.
    • Align labels at their left edge for a clean, scannable layout.

  4. Maintain Logical Order

    • Arrange options in a logical sequence (e.g., smallest to largest, alphabetical, or by relevance).

  5. Provide Help Text When Needed

    • Add brief help text or tooltips for complex options to provide additional context without cluttering the interface.

Examples

  • Selectable label vs. non-selectable label
  • Help text examples

Grouping and Alignment

  1. Use Clear Group Headings

    • Add headings or legends to describe the purpose of the group (e.g., "Choose Delivery Method").
    • Position headings above the first radio button and make them visually distinct (e.g., bold or larger font).

  2. List Radio Buttons Vertically

    • Vertical alignment improves readability, accessibility, and scalability across devices.
    • Use horizontal alignment only for very short labels and when space is limited.

  3. Logical Grouping and Visual Consistency

    • Avoid overwhelming users with too many options in one group.
    • For long lists, use subgrouping with headings or separators.
    • Use white space, dividers, or subtle background colors to differentiate groups.

  4. Avoid Nested Radio Buttons

    • Nested radio buttons increase complexity and reduce usability. Instead:
      • Use progressive disclosure to show/hide related options based on user selections.
      • Split complex decisions into separate, sequential questions.

Interaction Design

  1. Enable Only One Selection Per Group

    • Ensure only one radio button in a group can be selected at a time.

  2. Provide Instant Feedback

    • Radio buttons should respond immediately when clicked, with visual feedback (e.g., color change or animation).
    • Use sufficient contrast to distinguish selected and unselected states.

  3. Handling Situations Where No Selection Applies

    • Include options like "None," "Not applicable," or "Do not wish to provide" when necessary.

Default Selection Considerations

  1. When to Include a Default Selection

    • A common or recommended choice exists (e.g., "Standard Shipping").
    • A neutral or "safe" option is available.
    • Validation requires a selection to prevent errors.
    • The selected option continues an existing state or meets a legal/policy requirement.

  2. When Not to Include a Default Selection

    • To avoid bias in decisions requiring careful consideration (e.g., demographic data).
    • When the decision has significant consequences or could lead to accidental submission.
    • When no clear preference exists, or skipping the field is an option.

Radio Buttons in Tables

  1. Layout & Structure
    • Consistent Positioning: Place radio buttons in a dedicated, consistent column across all rows.
    • Visual Alignment: Align radio buttons perfectly within their cells, ensuring even spacing and clean vertical alignment.
    • Row Association: Use visual aids like zebra striping (alternating row colors) or subtle row highlighting on hover to associate radio buttons with their respective rows.
  2. Labeling & Headers
    • Clear Column Headers: Provide short, descriptive headers above each column that indicate the meaning of the radio button options (e.g., "Low," "Medium," "High").
    • Row Labels: Use clear labels in the leftmost column to identify each row. For long tables, consider making headers sticky.
    • Legend or Key: If a consistent scale is used across rows, include a legend directly above or below the table.
  3. Clarity & Context
    • Define Scope of Selection: Clearly communicate whether selections apply to the row, column, or entire table using clear headers and labels.
    • Visual Grouping: Use subtle borders or spacing to group radio buttons logically, ensuring users can distinguish between groups.

Example

Accessibility Considerations

  1. Use a Fieldset for Grouping
    • Wrap related radio buttons inside a <fieldset> and use a <legend> for the question or heading. 
    • This helps screen readers and improves visual organization.
  2. Check Your Labels
    • Use the 'for' attribute to associate label tags with radio buttons.
    • Ensure labels are readable with sufficient contrast and test usability with screen readers.
  3. Keyboard Navigation
    • Radio button groups should be accessible via the Tab key, with the entire group treated as a single tab stop.
    • Use Up/Down arrows to navigate vertical groups and Left/Right arrows for horizontal groups.
    • Pressing the Spacebar should select the currently focused option, ensuring compliance with accessibility standards.

See Accessibility Resources for more detailed guidance on implementing and testing accessible radio buttons.

Defaults for Input Controls 

Radio buttons are often used as input controls, which allow users to select an item and then perform an action on that item. In MyNSF, this is seen in many results tables. In most cases, a default option should not be selected when using radio buttons as input controls. This is to ensure that users are making an active choice before being able to perform an action.

Default Selection vs. No Default Selection

Listing Radio Button Options 

List Options Vertically

Ideally list options (button and label) should be arranged vertically with one option per line. Vertical arrangement increases the sustainability of the options and increases clarity between the radio button and the corresponding label. Horizontal listings can make it difficult to tell which label pertains to which radio button. If a horizontal layout with multiple options per line is used make sure the space between each makes it clear which choice goes with which label.

Horizontal List with Poor Spacing:

Vertical List with Good Spacing:

List options in a logical order

When applicable, list options should be listed in a logical order. This is applicable when the options exist on a scale or a continuum. For example you could list the options from: smallest to largest; oldest to newest; least likely to most likely, easy to difficult, etc.. Alphabetical order is appropriate when listing individual's names or when a list of terms is formalized (states, countries, etc.). If the terms are not formalized and users may be unsure of the terms, consider using other groupings other than alphabetical when listing options.

Keep Radio Button Options Unified

Radio buttons should always be unified and not broken up with subheadings. Subheadings or columned formatting gives the impression that there are selection groups, which should not be the case with radio buttons since users can select only one choice. If you find yourself needing subheadings or columns for radio options consider using checkboxes, a dropdown, or creating a new radio button field.

Example of good option unification:

Options Should Be Comprehensive and Clearly Distinct

Because radio buttons require exactly and only one choice, the options should be both comprehensive and clearly distinct. Radio button labels should be short and clear. The biggest usability problems for radio buttons come from labels that are vague, misleading, or describe options that are impossible for average users to understand. Don't offer options that overlap with other options.

If it's impossible to be comprehensive, offer a button labeled "Other," or something along those lines. If selected display a text field to type in his/her option.

Radio Buttons and Tables

Radio buttons are used within MyNSF in tables which display a large amount of information like search results tables. Radio buttons allow users to select a single item in a table row and perform an action on that item. In this case, no default selection should be made in the table, this prevents the user from taking actions by mistake. In some cases, using a default selection of 'none' might be more appropriate. In either case, action buttons should be disabled by default and only enabled once a selection has been made. 

Radio buttons should only be used when an action can only be performed when a selection is made and cannot be performed on multiple items. If a single action can be performed on multiple selections, use checkboxes. 

Example of radio buttons in a table

Maintain radio button selections while manipulating tables

Tables often have various functionality around filtering, sorting, and amount of results displayed. Radio button selections should always persist given any table functionality. For example, when a user makes a selection and then filters the table, the original selection should be maintained regardless if the original selection appears in the filtered results. With radio buttons, unlike checkboxes, since only one selection is possible there is little concern that the user perform an action on a previously selected and forgotten selection.


Progressive disclosure using Radio buttons

Progressive disclosure, which is displaying information that is most important initially, while displaying secondary information upon request, can help keep pages simple and easy to read. Therefore, when information is only needed when a particular radio button is selected, it may be beneficial to wait to display the secondary information until that radio button is selected. When a radio button is selected that needs additional input, the secondary information would display below the selected radio button. This ensures the relationship between the radio button selection and the secondary information is clear to the user. In addition, the secondary information should appear visually related on the screen, for example, by indenting it slightly, or adding a sub-heading.

If a radio button choice is changed, the secondary information related to that radio button would be hidden and any data not saved. If data will be lost, it is recommended that users be warned, and asked to confirm if they wish to change their selection.

Things to Avoid

Avoid Using Dropdowns when Radio Buttons are More Appropriate

Radio buttons have a lower cognitive load than dropdowns because all options are permanently visible which allows users to scan all available options without clicking (or typing) anything to reveal the information. Radio buttons are also easier to compare and easier to operate for users who have difficulty making precise mouse movements. This means that users typically take less time to make a selection using radio buttons versus dropdowns. Consider using radio buttons when ease of use is a primary focus and the screen space allows. Use dropdowns only if you don't have enough space to list the options, or if the additional cognitive load is not of great concern.  


Example of when to use radio buttons instead of dropdown

Avoid Nesting

Keep all options at the same level to avoid confusion. Avoid nesting radio buttons with other radio buttons or checkboxes. The introduction of subsets or hierarchy can confuse users' selection process since only one choice is allowed. Instead, consider creating conditional fields or fieldsets that are triggered to appear when a particular selection is made.  

Example of confusing nested radio buttons

Example of conditional / hidden second radio group


Avoid Using Radio Buttons as Actions, Especially when Changing Settings

When using radio buttons to modify settings, avoid using radio buttons as actions. Settings should never be considered as saved just by a radio button selection. A settings page should include an action button such as 'Ok' or 'Save' which confirms the radio button choice and modifies settings. If the user clicks the Back button without clicking to 'Save', any changes made to radio buttons on the page should be discarded and the original settings reinstated. The same guideline obviously holds if the user clicks a Cancel button. If the user first clicks Back and then Forward, then it's most appropriate to interpret this sequence as an Undo-Redo pair, meaning that the appearance of the controls should show the user's changes as if the user had never clicked Back. These changes should still not take effect on the backend until the user clicks "OK" or an equivalent command.

Example of radio buttons as selections and action buttons to 'confirm'

Exceptions


Exceptions Specific to Applications (Limited View)

Defaults for Input Controls in Tables

Typically, a default option should not be selected when using radio buttons as input controls. However, there are specific cases within MyNSF in which default selections are used due to technical limitations. One such example is for Duplicate Email Resolution for Participant Availability Requests. In this case, the radio button choice 'None' is selected as default. This helps satisfy the need for users to make an active choice and satisfies a technical limitation.
