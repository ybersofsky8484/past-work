Overview

The purpose of a button is to direct users to take an action. Buttons enable users to complete tasks (e.g., Save, Add), undo previous steps (e.g., Cancel), or advance through a process (e.g., Next). Buttons are critical interactive elements that should be:

  • Easy to recognize as buttons

  • Easy to find

  • Clearly communicate their intended function

  • Consistent in appearance across the interface

When to Use Buttons

Use buttons when:

  • The user is expected to take an action (e.g., Submit, Edit, Delete, Download)

  • The interaction results in a state change

  • The user needs to submit or confirm data

  • The user is triggering a specific action

  • The user is progressing through a workflow (e.g., clicking "Next" in a wizard)

  • Avoid overwhelming userslimit the number of buttons to clear, distinct actions

When to Use a Link Instead

Use links when:

  • Simply navigating between pages

  • Moving through the interface without changing state

  • Providing secondary or non-critical navigation

  • Returning to a previous page without submitting data

  • Linking to external websites or downloadable files

Usage

Button Types and Variants

Primary Buttons

  • Used for the main action the user is expected to take (e.g., Save, Continue, Submit)
  • Typically only one per view to reduce ambiguity
  • In rare cases, a second primary button may be justified (e.g., Search and Continue) if both are essential and clearly distinct. Example:

Secondary Buttons

  • Support less critical actions that complement or reverse the primary task (e.g., Close, Filter, Cancel).

  • Helps prevent unintentional clicks by reducing emphasis compared to primary actions

Example: Primary and Secondary Buttons

Split Buttons

A split button (sometimes called a dropdown button) combines a primary action with a list of secondary options. It allows users to either trigger the main action directly or select a related option from a dropdown. Refer to the Split Button standard <Replaces Buttons with Multiple Actions; need to update to the published link when the Split Buttons page is published> for more details. 

Button Design and Hierarchy

A clear visual hierarchy helps users quickly identify the most important action and understand how related actions are grouped together.

Primary Buttons

  • Should be given the highest visual priority to draw attention

Secondary Buttons

  • Should have a lower visual weight to avoid drawing focus or encouraging accidental clicks

Button Placement and Alignment

  • Be consistent across screens

  • Place buttons near the content they act on

  • Right-align buttons in modals and wizards (unless your application's style guide follows a different convention)

  • Left-align buttons in standard forms (unless your application's style guide follows a different convention)

Examples:

Grouping Buttons

  • When to group:
    • When they form an action pair with a clear contextual relationship (e.g., Save and Cancel)
    • If they are part of a workflow step sequence, such as in a wizard (e.g., Previous, Next)
  • When not to group:
    • If the buttons serve unrelated purposesin these cases, add more space between the groups than between related buttons.
    • When one is a destructive action (e.g., Delete, Remove, Uninstall) and the other is non-destructive (e.g., Save, Download, Edit). This can lead to accidental clicks and user frustration.
    • Example:

Use of Whitespace

  • Use spacing to avoid clutter

  • Maintain consistent spacing for legibility and focus

Avoid Too Many Buttons

  • Keep choices limited and meaningful

  • Having too many choices leads to confusion

  • Consider using a split button if multiple actions are possible <update to published link when available>

Button Labels

A button is a call-to-action. When deciding on the label for a button you should ask, What is my user trying to do? 

Do

  • Use sentence case for readability (e.g., "Yes, complete this task" not "Yes, Complete This Task")
  • Lead with a strong verb (e.g., "File a complaint" not "Complaint filing")
  • Keep labels short but meaningful
  • Label actions consistently across pages

Don't

  • Use vague labels like "Click Here"
  • Use ALL CAPS for button labels

Icons in Buttons

  • Use to reinforce meaning, not replace text
  • Keep icon use minimal
  • Avoid icons that require interpretation or cultural knowledge
  • Refer to Icons standard for additional guidance

Button States

Users should easily distinguish between different button states.

Default

  • Clearly styled as clickable

  • Uses unambiguous, actionable language

Focused

  • Visual change on keyboard tab or hover

  • Must display a visible focus state for accessibility

Disabled

  • Unavailable but potentially valid action (e.g., 'Next' when fields are incomplete)

  • Must include title/hover/screen reader text explaining how to make the button active

  • Should be visually distinct from both primary and secondary buttons

  • Hide buttons entirely if users lack permission to perform the action, rather than disabling them

  • Avoid leaving users guessing why a button is disabled

In-Progress

When a button triggers an action with a noticeable delay, provide immediate visual feedback by changing the button's state to "in progress" until the process complete:

  • Show a spinner icon for actions taking 25 seconds

  • Update the button label (e.g., change "Save" to "Saving") to reflect the current state

  • Temporarily disable the button and any related or dependent controls to prevent duplicate submissions

  • For actions longer than 5 seconds, consider using dedicated progress indicators (e.g., progress bars or modals)

Special Use Cases (Not Ideal, But Acceptable)

Sidebar buttons in PSM are present on the main proposal page (example). Some of the sidebar buttons do initiate an action directly, such as Print. However, others launch a modal or navigate into a wizard or separate page where the actions can be taken. Buttons were used instead of links due to the following:

  • Two of these - Share Proposal with SPO/AOR (goes to another page), and Initiate Proposal Submission (goes into a wizard), are primary actions on the page and needed to stand out more than an action like 'Print'
  • Visually it appeared awkward to display some as links and some as buttons

In MyNSF, there is a 'Send Email' button that links to a preview screen where you can then send the email.

  • UX Recommendation:  Change button label to 'Preview Email'

Non-Compliance Instances (For Internal Reference)

Within Awards, there is an instance where 'Save' is shown as a primary action button and is placed on the far left of the screen.   

  • UX Recommendations:  'Save' should be a secondary action button (since Process automatically saves), and it should be moved to the right side of the page.
